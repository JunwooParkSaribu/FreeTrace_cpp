# FreeTrace C++ Tracking: Achieving 100% Match with Python

## Goal
Achieve 100% identical trajectory output between C++ and Python tracking on all 7 samples with `jump_threshold=10.0`.

## Starting State
- Tracking match rate was ~73.9% on sample 0 (jump=10)
- Greedy iteration divergence at iteration 163 within subgraph 44
- C++ found 2 paths while Python found 3 after identical greedy selections

## Root Cause Investigation

### Problem 1: CPython Set Iteration Order (carried from prior sessions)
Python's `set()` uses hash-table slot order with **perturbation probing AND linear probing** (LINEAR_PROBES=9). The function `nx.descendants_at_distance(G, source, 1)` returns a set, and DFS path enumeration iterates this set.

Three bugs were fixed in `cpython_set_order()`:
1. **Perturb shift order**: `perturb >>= PERTURB_SHIFT` was happening AFTER computing next slot instead of BEFORE
2. **Missing LINEAR_PROBES**: CPython's `set_add_entry` checks 9 consecutive slots after collision before perturbation — this was completely missing
3. **Resize condition**: Used `table_size * 3` instead of CPython's `mask * 3` (where `mask = table_size - 1`)

### Problem 2: Orphan Reconnection Iteration Order (fixed this session)
**File**: `src/tracking.cpp`, `greedy_assign_pass()`, orphan reconnection block

After removing selected trajectory nodes from the subgraph, orphaned nodes (unreachable from source) get reconnected with a direct edge from source. The **iteration order** over nodes matters because reconnecting node A to source can make node B reachable via A, changing whether B also gets a direct source edge.

- **C++ (bug)**: Iterated over `std::set<Node>(sub_graph.get_nodes())` — sorted order `(time, idx)`
- **Python**: Iterated over `list(sub_graph.nodes)` — **insertion order** (NetworkX preserves insertion order)

**Fix**: Changed to `sub_graph.get_nodes_ordered()` which returns nodes in insertion order, matching Python.

### Problem 3: Phase 2 Orphan Reconnection (fixed this session)
**File**: `src/tracking.cpp`, `greedy_assign_pass()`

The C++ code used a single `greedy_assign_pass()` function for both phase 1 (exploring NB_TO_OPTIMUM=4 starting indices) and phase 2 (final assignment with best starting index). Both phases included orphan reconnection.

However, **Python's phase 2 does NOT have orphan reconnection** — only phase 1 does.

**Fix**: Added `if (debug_phase1)` guard around the orphan reconnection block. The `debug_phase1` parameter was already `true` for phase 1 and `false` for phase 2 calls.

### Root Cause of non-100% results
Investigated and confirmed: the 1-3 point differences at non-default jump thresholds come from **inherent floating-point precision differences** in cost function computation (~4.55e-13, ~1.3 ULP). Example: cost `338.61922773390955` (C++) vs `338.61922773391001` (Python). This is due to different FP operation ordering in `predict_cauchy_tracking`/`predict_long_seq` and is not practically fixable.

## Key Technical Concepts for Future Reference
- **CPython set probing**: initial slot = `hash & mask`, then LINEAR_PROBES=9 consecutive slots, then `perturb >>= 5; i = (i*5 + perturb + 1) & mask`
- **CPython set resize**: when `fill * 5 >= mask * 3` (mask = table_size - 1)
- **CPython tuple hash**: xxHash-based with XXPRIME_1/2/5 constants, rotate-multiply accumulator
- **NetworkX node order**: insertion order preserved (Python 3.7+ dict ordering)
- **x86-simd-sort**: Intel's AVX2 sort matching numpy's `np.argsort(kind='quicksort')` — integrated in `include/`
